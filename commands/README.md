# `commands/` — Canonical slash-command sources

This directory holds the **single source of truth** for every Cadenza slash command. Each `<name>.md` file becomes a slash command in all four supported agent tools via per-tool bridges generated by [`scripts/commands-sync.sh`](../scripts/commands-sync.sh) (and, for Codex, a one-shot install via [`scripts/install-codex-commands.sh`](../scripts/install-codex-commands.sh)).

> **Looking for the list of currently available commands?** → [`INDEX.md`](./INDEX.md).
> This README covers the *format contract and bridge mechanics*; INDEX.md is the per-command catalog with descriptions, arguments, and invocation syntax across tools.

## Why canonical here rather than in `.claude/commands/`?

Three reasons:

1. `.claude/commands/` is Claude Code-specific. Using it as the source ties our naming to one vendor.
2. Gemini CLI requires TOML, not markdown. We need a build step anyway; moving the source out of any one vendor's directory makes that step less weird.
3. Codex cannot read project-level prompt directories today ([openai/codex#7480](https://github.com/openai/codex/issues/7480)). We want our project-internal layout independent of Codex's limitation.

## File format

Use **markdown with YAML frontmatter**, the format shared by Claude Code and Codex. Example:

```markdown
---
description: "One-line description that shows in /help"
argument-hint: "[optional-args]"
allowed-tools: Bash(pnpm *), Read, Grep
---

The prompt body. Use $ARGUMENTS for positional args (captures everything after the command name) or $1, $2, ... for specific positions.

If the command should do X, describe exactly what X is. Be imperative, not conversational.
```

### Supported frontmatter fields

- **`description`** — used by Claude Code, Codex, and Gemini. Shown in `/help`; required (the sync script supplies a generic default if missing).
- **`argument-hint`** — used by Claude Code and Codex. Gemini lacks an equivalent; ignored there.
- **`allowed-tools`** — used by Claude Code to pre-approve tool calls. Ignored by Codex and Gemini.
- **`model`** — used by Claude Code only. Rarely useful; avoid unless you have a specific reason.

### Argument placeholders

- `$ARGUMENTS` — everything after the command name (Claude Code + Codex syntax).
- `$1`, `$2`, ... — positional (Claude Code + Codex).
- In Gemini's TOML, the sync script rewrites these to `{{args}}` (Gemini doesn't support positional variants; you lose granularity on Gemini).

## Invocation cheat sheet

After a command named `foo.md` is synced:

| Tool        | Command                  | Reads / resolves                            |
| :---------- | :----------------------- | :------------------------------------------ |
| Claude Code | `/foo`                   | `.claude/commands/foo.md` (symlink)         |
| OpenCode    | `/project:foo`           | `.claude/commands/foo.md` (Claude fallback) |
| Codex       | `/prompts:cadenza-foo`   | `~/.codex/prompts/cadenza-foo.md` (symlink) |
| Gemini CLI  | `/foo`                   | `.gemini/commands/foo.toml` (generated)     |

Details: OpenCode's Claude Code fallback is documented at <https://opencode.ai/docs/rules/>. Codex requires `scripts/install-codex-commands.sh` to be run once per developer (prompts are user-scope only); the `cadenza-` prefix prevents cross-project collisions.

## Adding a new command

1. Create `commands/<name>.md` with frontmatter and body.
2. Run `scripts/commands-sync.sh` (or `pnpm commands:sync` once `package.json` exists). This regenerates `.claude/commands/*.md` symlinks and `.gemini/commands/*.toml` artifacts.
3. Commit all of: `commands/<name>.md`, the `.claude/commands/<name>.md` symlink, and the `.gemini/commands/<name>.toml` artifact.
4. For Codex users: they rerun `scripts/install-codex-commands.sh` once to pick up the new file.

## Keeping them cross-portable

- **Stick to vanilla markdown in the body.** Avoid Gemini-only features like `!{shell-cmd}` and `@{file-path}` injection — they don't exist in Claude Code or Codex.
- **Keep commands short and focused.** Under ~500 words is a good target (matches Claude skills best-practice).
- **Use imperative verbs.** "Read X, summarize Y, return a table with columns Z" beats conversational prompts.
- **Reference files by relative path from repo root.** Do not use `~/` or `$HOME`; those mean different things in different tools.
